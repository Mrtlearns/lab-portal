# Exercise 25 — CUA User Monitoring & Reporting

> [!NOTE]
> Rewritten for A4H (ABAP 2023 Docker)

**Scenario**

Now that users and roles are distributed across CUA-connected child clients, you will use the central monitoring tools to audit user activity, identify locked accounts, and generate a cross-system user report for the training environment.

Prerequisites:
- **Central system:** A4H client 001
- **Child system:** A4H client 000
- **Exercise 23 & 24 completed** — `Z_TRAIN_USER` exists and `Z_ENDUSER_ROLE` is distributed

## Task 1: Display the central user overview

1. Log on to the **central system** (A4H client 001) as `TRAIN-01`.
2. Call transaction `SCUM`.
3. Navigate to **User Administration → Central User List**.
4. Leave all filters blank and choose **Execute** to list all CUA-managed users.
5. Locate `Z_TRAIN_USER` in the list and note its **Status** and **Last Logon** columns.
6. Record: is the user currently **Active**, **Locked**, or **Never Logged On**?

## Task 2: Check for locked users across child systems

1. Still in `SCUM`, choose **Tools → Monitoring → Locked Users**.
2. Choose **Execute** to run the report across all connected child systems.
3. Review the results:
   - Identify any users locked in child system `A4H-000`.
   - Note whether the lock originated in the **central** or **local** system (check the **Lock Reason** column).
4. If `Z_TRAIN_USER` appears locked, select it and choose **Unlock** to demonstrate central unlock capability.

## Task 3: Generate a user-role assignment report

1. Call transaction `SUIM`.
2. Under **Users**, select **Users by Complex Selection Criteria → By Roles**.
3. In the **Role** field, enter `Z_ENDUSER_ROLE`.
4. Choose **Execute**.
5. Confirm `Z_TRAIN_USER` appears in the results for both the central and child systems.
6. Export the list to a local file using **List → Export → Spreadsheet** and save it as `CUA_Role_Report.xlsx`.

## Task 4: Audit logon failures (optional bonus)

1. Call transaction `SM20` (Security Audit Log).
2. Set **From date** to today and filter on **Failed Logon Attempts**.
3. Choose **Execute**.
4. Review any failed logon entries for `Z_TRAIN_USER`.
5. Discuss with your instructor: what is the default **max failed logon** threshold in the A4H system, and where is it configured?

> [!TIP]
> The parameter `login/fails_to_user_lock` in transaction `RZ10` controls how many failed attempts lock an account. The default is 6.
