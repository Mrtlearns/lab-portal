## Rewritten for A4H (ABAP 2023 Docker)

### Scenario
Use the local A4H two-client CUA-style setup from the previous exercises:

- **Central:** A4H client **001**
- **Child:** A4H client **000**

Create a role in the child, assign it centrally to a new user, validate distribution, and observe the restricted maintenance behavior in the child.

---

## Task 1: Create an end-user role in the child client
1. Log on to **A4H client 000**.
2. Start `PFCG`.
3. Copy role `SAP_BC_ENDUSER` to `Z_SAP_BC_ENDUSER`.
4. Open **Authorizations** and generate the profile.
5. Save the role.

---

## Task 2: Create a centrally administered user for the child
1. Log on to **A4H client 001**.
2. Start `SU01`.
3. Create user `CHILD##`.
4. Maintain basic data:
   - last name
   - initial password
   - optional user parameters such as `BUK = 42`
5. On the **Roles** tab, assign the role **`Z_SAP_BC_ENDUSER`** for receiving system **`A4HCLNT000`**.
6. Save.

> If the child role text is not selectable centrally, run the text comparison in child client `000` via `PFCG` â†’ **Environment â†’ Text Comparison for CUA Central System**, then retry.

---

## Task 3: Check the central distribution log
1. In **client 001**, keep `CHILD##` selected in `SU01`.
2. Choose **Environment â†’ Distribution log**.
3. Include **Successful** on the Users tab.
4. Execute and confirm green/successful distribution entries.

---

## Task 4: Validate the user in the child client
1. Log on to **A4H client 000**.
2. Start `SU01` and display `CHILD##`.
3. Review the tabs.
4. Switch to change mode and observe that only the fields allowed by your `SCUM` settings remain editable locally.

---

## Task 5: Log on as the child user and maintain own data
1. Log on to **A4H client 000** as `CHILD##`.
2. Start `SU3` or use **System â†’ User Profile â†’ Own Data**.
3. Modify only the fields still available for local maintenance.
4. Save.

---

## Task 6: Test central logon behavior
1. Try to log on to **A4H client 001** as `CHILD##`.
2. In a classic CUA model, a user created only for the child does **not** become a locally valid end user in the central client unless a system assignment for the central client exists.
3. Record the resulting behavior in your system.

---

## Task 7: Cross-system role reporting
1. In **A4H client 001**, start `SUIM`.
2. Navigate to **User â†’ Cross-System Information (Central User Administration)**.
3. Execute **Users by Roles** for user `CHILD##`.

## Expected result
You have proven the core CUA runtime behavior in A4H:
